## 1. توحيد مسار رفع صور المعرفة المرئية

### المشكلة
كانت توجد مساران لرفع صور المنيو والمنتجات داخل قاعدة المعرفة:
- **التاب الأول (مكرر):** تاب "صورة / منيو" داخل نموذج إضافة المعرفة (`activeTab === "image"`) → يرسل لـ `/api/knowledge` → يستخرج النص فقط دون حفظ الصورة أو إنشاء `KnowledgeAsset`.
- **القسم الثاني (الصحيح):** قسم "المعرفة المرئية" أسفل النموذج → يرسل لـ `/api/knowledge/assets` → ينشئ `KnowledgeAsset` + `KnowledgeDocument` مرتبطان ببعض.

### الفرق الجوهري
القسم الصحيح (assets) يحفظ الصورة ويربطها بالمعرفة النصية ثنائي الاتجاه، مما يتيح للبوت إرسال الصورة للعميل. التاب المكرر لا يحفظ رابط الصورة في قاعدة المعرفة.

### الحل
- **حُذف تاب "صورة / منيو"** من المصفوفة `tabs` داخل `KnowledgeManager`.
- **الواجهة الباقية:** قسم "المعرفة المرئية" الكامل (`Visual Knowledge`) الذي يدعم: منيو / عروض / منتجات، مع تحليل AI وحفظ الصورة وربطها بالمعرفة.

### الملفات المعدّلة
- `src/components/dashboard/knowledge-manager.tsx` — حذف `{ id: "image", ... }` من مصفوفة `tabs`، وإزالة منطق `activeTab === "image"`.

## 2. طبقة فهم الوسائط الموحدة

### Architecture

```
src/lib/media-understanding.ts
```

طبقة خدمة مركزية مسؤولة عن:

| الوظيفة | الدالة |
|---|---|
| تحليل الصور | `analyzeImageFromUrl(url, mimeType)` |
| تفريغ الصوت | `transcribeAudioFromUrl(url, mimeType)` |
| معالجة رسالة كاملة | `understandMessageMedia(messageId, tenantId)` |
| بناء سياق AI | `buildMediaContextForAi(understanding)` |

**النوع الموحد للمخرجات:**
```typescript
interface MediaUnderstandingResult {
  type: "image" | "audio";
  url: string;
  contentHash?: string;        // SHA-256 للمحتوى (للتكامل المستقبلي)
  understanding: string;       // الوصف / النص المستخرج
  analyzedAt: string;
  provider: string;
  model?: string;              // للصوت: اسم نموذج Whisper
  error?: string;              // لا يوقف التدفق، يُسجل فقط
}
```

**التخزين:** يُحفظ في `message.metadata.mediaUnderstanding` (التحديث آمن — لا يكسر الرسالة).

**حماية SSRF:** التحقق من عناوين IP الخاصة قبل أي fetch خارجي.

### الملفات الجديدة
- `src/lib/media-understanding.ts` — الطبقة الكاملة

## 3. فهم الصور داخل المحادثات

### التدفق

```
Incoming Message (image attachment)
  → ingressProcessor: save message + enqueue media-understanding-queue
  → media-understanding-worker: analyzeImageFromUrl() via AI providers
  → Message.metadata.mediaUnderstanding = { images: [...] }
  → ai-worker: buildMediaContextForAi() → injects into AI context
  → generateAiReply() → bot responds with image-aware context
```

**مزود التحليل:** يستخدم نفس `AiProvider` المُعدّ في النظام (Gemini / OpenAI / OpenRouter) مع fallover تلقائي بين المزودين.

**الأولوية:** تحليل الصورة يُجدوَل بـ `priority: 2` في القائمة، لكنه لا يُعيق رد AI. إذا اكتمل التحليل قبل وصول دور AI، يستخدمه. وإلا ينتقل لـ fallback النصي.

**MIME types المدعومة:** `image/jpeg`, `image/png`, `image/webp`, `image/gif`, `image/bmp`

**الحجم الأقصى:** قابل للإعداد عبر `MEDIA_UNDERSTANDING_MAX_IMAGE_MB` (افتراضي: 10 MB)

### الملفات المعدّلة
- `src/server/channels/ingressProcessor.ts` — إضافة `mediaUnderstandingQueue.add()` بعد حفظ الرسالة
- `workers/ai-worker.ts` — قراءة `metadata.mediaUnderstanding` وحقنه في context

## 4. فهم التسجيلات الصوتية (Whisper)

### إعدادات السوبر أدمن

```
POST /api/admin/speech-settings
GET  /api/admin/speech-settings
```

**الحقول:**
| الحقل | النوع | الوصف |
|---|---|---|
| `enabled` | boolean | تفعيل التفريغ الصوتي |
| `provider` | "openai" | المزود (حالياً OpenAI فقط) |
| `apiKey` | string | مفتاح OpenAI (مُشفَّر بـ AES-256-GCM) |
| `transcriptionModel` | string | نموذج Whisper (افتراضي: `whisper-1`) |
| `maxAudioSizeMB` | number | الحد الأقصى لحجم الملف (1-100 MB) |
| `allowedMimeTypes` | string[] | أنواع الصوت المسموحة |
| `language` | string | "auto" أو كود ISO-639-1 |

**الأمان:** المفتاح يُشفَّر بنفس آلية `AI_KEY_ENCRYPTION_SECRET` / `NEXTAUTH_SECRET` المستخدمة في باقي المفاتيح. لا يُعاد إرساله للواجهة أبداً (يُعاد `apiKeyConfigured: boolean` فقط).

### التدفق

```
Incoming voice message
  → ingressProcessor: enqueue media-understanding-queue
  → media-understanding-worker: transcribeAudioFromUrl()
    → SpeechSetting (enabled + apiKeyEncrypted)
    → OpenAI Whisper API (whisper-1)
    → transcript stored in message.metadata.mediaUnderstanding.audios
  → ai-worker: voice transcript injected as: 'Voice transcription: "..."'
```

**Fallback عند الفشل:** الخطأ يُسجَّل، والمحادثة تستمر بالوصف النصي العادي للمرفق.

### الملفات الجديدة
- `src/lib/models/speech-setting.ts` — نموذج الإعدادات
- `src/app/api/admin/speech-settings/route.ts` — GET/POST/DELETE للسوبر أدمن

## 5. عزل وكيل الدعم

### الحالة الحالية (موثّقة)

تم مراجعة الكود وتأكيد أن العزل مُطبَّق بشكل صحيح في جميع طبقات النظام:

| الطبقة | الملف | الحالة |
|---|---|---|
| قائمة المحادثات | `GET /api/inbox/conversations` | ✅ `shouldScopeToAssignedConversations` |
| تفاصيل محادثة | `GET /api/inbox/conversations/[id]` | ✅ `assignedOnly` filter |
| إرسال رد | `POST /api/inbox/conversations/[id]/reply` | ✅ `assignedOnly` filter |
| إضافة ملاحظة | `POST /api/inbox/conversations/[id]/notes` | ✅ `assignedOnly` filter |
| تحديث الحالة | `PATCH /api/inbox/conversations/[id]/status` | ✅ يتطلب `inboxManage` (لا يملكه agent) |
| Realtime Stream | `GET /api/inbox/stream` | ✅ `assignedOnly` snapshot + `sync.required` للأحداث |
| Analytics | داخل `inbox/service.ts` | ✅ `assignedAccessFilter` مُطبَّق |

**آلية Realtime:** عند `assignedOnly === true`، يرسل الـ Stream حدث `sync.required` بدلاً من بث حدث المحادثة مباشرةً، مما يُجبر الـ Frontend على إعادة جلب البيانات المفلترة. هذا يمنع تسريب بيانات محادثات غير مسندة.

**صلاحيات Agent:**
```typescript
agent: [contactsRead, companiesRead, inboxRead, inboxReply, ticketsRead]
// ✅ لا يملك: inboxManage, inboxAssign, knowledgeManage, settingsManage
```

## الأداء والتوسع

- **Queue-based processing:** تحليل الوسائط يعمل في queue مستقل (`media-understanding-queue`) لا يعوق رد AI.
- **حجم محدود:** MAX_IMAGE_BYTES و MAX_AUDIO_BYTES تمنع تحميل ملفات كبيرة للذاكرة.
- **Content Hash:** كل نتيجة تحمل `contentHash` (SHA-256) للتمييز المستقبلي وتجنب تحليل نفس المرفق مرتين.
- **SSRF Protection:** التحقق من عناوين IP الخاصة قبل أي fetch.
- **Timeout:** كل fetch له حد زمني (`MEDIA_UNDERSTANDING_FETCH_TIMEOUT_MS`، افتراضي 15 ثانية).
- **Concurrency:** worker مستقل بـ concurrency قابل للإعداد (`MEDIA_UNDERSTANDING_WORKER_CONCURRENCY`، افتراضي 5).

## الحماية والأمان

- مفتاح OpenAI للصوت مُشفَّر بـ AES-256-GCM، لا يُرسل للواجهة.
- التحقق من MIME type للصور والصوت قبل المعالجة.
- SSRF protection على روابط المرفقات (رفض IPs الخاصة والـ localhost).
- tenant scope محفوظ في كل عمليات قاعدة المعرفة، القنوات، والمحادثات.

## التوافر (Availability)

- فشل تحليل الصورة أو الصوت لا يوقف المحادثة.
- عند فشل مزود AI للصور: ينتقل للمزود التالي (fallover على `AiProvider` المرتبة بـ `priority`).
- عند فشل Whisper: يُسجَّل الخطأ والمحادثة تستمر بالوصف النصي للمرفق.

- كل error في `understandMessageMedia` يُلتقَط داخل try/catch per-attachment.
