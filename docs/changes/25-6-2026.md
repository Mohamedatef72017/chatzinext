## 11:30 AM — planSnapshot Strategy (ADR-001)

### Architecture Changes
- تحديد ADR-001: الاشتراكات بحالة `{ active, trialing, past_due }` تقرأ الميزات من `TenantSubscription.planSnapshot.features[]` — وليس من `BillingPlan` الحي. يُحدَّث الـ snapshot فقط عند الـ checkout وعند كل تجديد. تعديلات الأدمن على خطة ما تسري على دورة الفوترة التالية، وليس في منتصف الدورة الحالية.

### Backend Changes

**`src/lib/billing/plan-resolver.ts`**

الوحدة المركزية للـ billing engine. تُنفّذ استراتيجية الـ snapshot-first مع أولوية resolution واضحة:

```
Priority (highest → lowest):
1. Entitlement override (collection Entitlement، isOverride: true)
2. TenantSubscription.planSnapshot.features[] (للاشتراكات النشطة)
3. BillingPlan.features[] (للخطة المجانية أو عند غياب الـ snapshot)
4. Safe default (disabled / 0)
```

- حذف `populate("planId")` — لم يعد مطلوباً في المسار الأساسي.
- إضافة snapshot-first resolution: إذا وُجد `subscription.planSnapshot.features` والاشتراك active/trialing/past_due، اقرأ الميزات مباشرة من الـ snapshot المُضمَّن بدون أي DB query إضافي.
- إضافة fallback للـ `BillingPlan.findById()` الحي عند غياب الـ snapshot (مستأجرو الخطة المجانية، الاشتراكات الملغاة، checkouts جديدة قبل كتابة أول snapshot).
- إضافة `resolvedFrom: "snapshot" | "live_plan" | "default"` إلى `ResolvedPlan` — مرئي في debug/audit contexts.
- إضافة `source: "snapshot"` إلى `ResolvedFeature.source`.
- جميع المسارات الثلاثة تضبط المتغيرات المحلية `planId, planName, planSlug, resolvedFrom` قبل تطبيق الـ overrides.

**الدوال المُصدَّرة من `plan-resolver.ts`:**

| الدالة | الوصف |
|---|---|
| `resolveTenantPlan(tenantId)` | يُعيد `ResolvedPlan` كاملاً (جميع الميزات + metadata) |
| `resolveFeature(tenantId, featureKey)` | يُعيد `ResolvedFeature` لمفتاح واحد |
| `setEntitlementOverride(tenantId, key, value, expiresAt?)` | يكتب override لمستأجر محدد |

**`src/lib/billing.ts`** (`handleInvoicePaymentSucceeded`)
- عند كل `invoice.payment_succeeded` مع `billing_reason = subscription_cycle`، يقرأ `BillingPlan` الحالي ويكتب snapshot جديد في `TenantSubscription.planSnapshot`.
- كتابة الـ snapshot atomic مع إعادة تعيين `usedMessages` في نفس `updateOne`.
- **التأثير**: تغييرات الأسعار والميزات التي يُجريها الأدمن مضمونة التفعيل في الدورة التالية. لا تغييرات رجعية على الاشتراكات الجارية.

### Database Changes
- لا إضافات في الـ schema — حقل `planSnapshot` موجود بالفعل على `TenantSubscription`. هذا التغيير يُوصّل مسار القراءة لأول مرة.

### Performance Improvements
- المسار snapshot-first يستبدل `populate("planId")` join بقراءة مباشرة من embedded document — يُلغي DB round-trip واحداً لكل استدعاء `resolveTenantPlan()` في حالة الاشتراك النشط.

#### `usage-engine.ts`
محرك قياس الاستخدام العام. يستبدل منطق AI-message-only في `quota.ts`.

**Hot Path (Redis):**
```
[API Request]
     ↓
reserveUsage(tenantId, featureKey, amount)
     ↓
Redis SET NX (bootstrap from MongoDB عند أول استخدام في الفترة)
     ↓
Redis INCRBY (atomic)
     ↓
Check: newCount > limit? → DECRBY + throw
     ↓
كل 10 عمليات: syncToMongo (fire-and-forget)
```

**Redis Key Format:**
```
usage:{featureKey}:{tenantId}:{periodKey}
مثال: usage:ai_messages:64abc...:2026-06
```

**Period Keys:**
- `monthly` → `YYYY-MM`
- `yearly` → `YYYY`
- `never` → `all`

**Failure Modes:**

| السيناريو | السلوك |
|---|---|
| Redis down + خطة مجانية (limit ≤ 200) | Fail closed — اقرأ MongoDB، احجب إذا تجاوز الحد |
| Redis down + خطة مدفوعة | Fail open — سجّل خطأ، اسمح بالطلب |
| MongoDB sync يفشل | تحذير في اللوج، استمر (Redis هو المصدر) |

**خاصية Idempotency Key:**
```typescript
await reserveUsage(tenantId, "ai_messages", 1, {
  idempotencyKey: messageId
});
// آمن للاستدعاء المتعدد — التسجيل يحدث مرة واحدة فقط
```

**الدوال:**
- `reserveUsage(tenantId, featureKey, amount, options)` — الحجز الذري
- `getFeatureUsage(tenantId, featureKey)` — حالة الاستخدام الحالية
- `flushUsageToMongo(tenantId, featureKey)` — sync فوري للـ MongoDB
- `resetUsageForPeriod(tenantId, featureKey, periodKey)` — إعادة ضبط الفترة
- `getFullUsageSnapshot(tenantId)` — snapshot كامل لجميع الميزات

## 01:00 PM — Trial & Grace Period Lifecycle Cron

### Architecture Changes
- نقل تطبيق انتهاء التجارب وفترات السماح من المسار الساخن لطلبات API إلى HTTP cron endpoint مستقل، آمن للجدولة الخارجية بأي cron service يدعم HTTP (Vercel Cron، Upstash، render.com، إلخ).

### Backend Changes

**`src/app/api/cron/billing-lifecycle/route.ts`** (ملف جديد)

**`POST /api/cron/billing-lifecycle`** — تُنفّذ مرورَين idempotent:

**المرور الأول (Trial Expiry):**
```
ابحث عن: { status: "trialing", trialEndsAt: { $lte: now } }
لكل اشتراك:
  ↓
  إذا وُجد stripeSubscriptionId:
    استعلم Stripe → active? → "active" | غيره → "canceled"
  ↓
  بدون stripeSubscriptionId (لا payment method):
    → "canceled"
  ↓
  updateOne الحالة
  ↓
  appendSubscriptionHistory (trial_converted | trial_expired)
  ↓
  writeBillingAudit (trial.converted | trial.expired)
```

**المرور الثاني (Grace Expiry):**
```
ابحث عن: { status: "past_due", graceEndsAt: { $lte: now } }
لكل اشتراك:
  ↓
  إذا وُجد stripeSubscriptionId:
    cancelSubscription immediately (best-effort، استمر عند الخطأ)
  ↓
  updateOne → { status: "canceled", currentPeriodEnd: now }
  ↓
  appendSubscriptionHistory (grace_expired)
  ↓
  writeBillingAudit (grace.expired)
```

**`GET /api/cron/billing-lifecycle`** — يُعيد `{ pendingTrials, pendingGrace }` للـ monitoring dashboards.

**الأمان:**
- مؤمَّن بـ `CRON_SECRET` env var.
- يقبل كلا الهيدرين: `Authorization: Bearer <secret>` و`x-cron-secret: <secret>`.
- يُعيد 401 بدون secret صالح.
- إذا لم يُضبط `CRON_SECRET`، كل الطلبات تُرفض.

**Idempotency:**
- آمن للتشغيل المتعدد — الاشتراكات المُعالجة بالفعل تُتخطّى بفضل query الحالة+التاريخ.

**Response:**
```json
{
  "ok": true,
  "processedAt": "2026-06-25T13:00:00.000Z",
  "trialExpired": 3,
  "graceExpired": 1,
  "errors": 0
}
```

### New Environment Variables
```
CRON_SECRET=<random_string_min_32_chars>
```

### Recommended Cron Configuration (`vercel.json`)
```json
{
  "crons": [
    { "path": "/api/cron/billing-lifecycle", "schedule": "0 */6 * * *" }
  ]
}
```

### `src/lib/ai/build-system-prompt.ts` — قسم RESPONSE FORMATTING الجديد

أُضيف بعد قسم RESPONSE QUALITY مباشرةً. يُلزم النموذج بـ 5 قواعد واضحة:

**SEPARATE YOUR IDEAS:**
كل فكرة مستقلة في فقرتها الخاصة مع سطر فارغ. ممنوع دمج أفكار متعددة في كتلة نص واحدة.

**USE LISTS WHEN APPROPRIATE:**
- 3 عناصر أو أكثر (خدمات، أسعار، خطوات، خيارات) → قائمة bullet point، عنصر واحد لكل سطر
- تسلسل خطوات يتبعها العميل → قائمة مُرقَّمة (1. → 2. → 3.)
- ممنوع ضغط القائمة في جملة واحدة بفواصل

**KEEP PARAGRAPHS SHORT:**
2–3 جمل لكل فقرة كحدٍّ أقصى. إذا احتاج الموضوع تفصيلاً، يُقسَّم إلى فقرات قصيرة متعددة.

**NEVER SEND A WALL OF TEXT:**
إذا تجاوز الرد 4 أسطر بدون سطر فارغ → كثيف جداً، اقسمه.

**DO NOT USE MARKDOWN DOCUMENT SYNTAX:**
ممنوع ## أو ### headers أو horizontal rules. هذه واجهة محادثة وليست مستنداً.

```typescript

  const intent = inputData.businessIntent as BusinessIntent;

      "intentSignal=purchase_ready → customer is close to booking or buying; " +
```

## Risks / Notes

1. **`PLAN_DEFAULTS_FALLBACK` removed** — مستأجرو `BillingPlan` بدون `features[]` يتلقون feature map فارغاً (كل فحوصات الـ entitlement تُرفض). **يجب** تشغيل `scripts/seed-plan-features.ts` على قاعدة بيانات الإنتاج قبل النشر.
2. **`CRON_SECRET` مطلوب** — إذا لم يُضبط، يُعيد cron endpoint 401 على كل طلب. اضبطه قبل توصيل الجدولة الخارجية.
3. **Prompt cache cold start** — أول AI reply بعد النشر أو بعد انتهاء TTL يُعيد بناء الـ prompt. هذا سلوك متوقع وليس خطأً.
4. **Stripe best-effort cancel عند انتهاء grace** — إذا أعاد Stripe خطأً (مثلاً الاشتراك ملغى بالفعل على Stripe)، يستمر الـ cron ويُعلّم السجل المحلي `canceled`. لا retry mechanism لاستدعاء Stripe؛ تنسيق حالة Stripe والمحلية يحدث عبر webhook التالي.
5. **`resolvedFrom` قيمة union جديدة** — أي TypeScript consumer لـ `ResolvedPlan` يستخدم exhaustive type check يحتاج تحديثاً؛ `"snapshot"` هو الآن عضو في الاتحاد.
6. **GLOBAL_CRM_SYSTEM_PROMPT زاد بشكل ملحوظ** — ~200 سطر بعد إضافة RESPONSE FORMATTING مقابل 73 سابقاً. يؤدي إلى استهلاك أعلى قليلاً من الـ tokens في كل استدعاء Mastra. مبرر: التحسين الجوهري في جودة السلوك. Prompt cache يُخفف الأثر على الأداء.
7. **Output token limit للـ agent: 400 → 600** — زيادة طفيفة في تكلفة tokens per message. مبرر: الردود المعقدة كانت تُقطع عند 400.
8. **intentSignal لا يُعاد حقنه في `unifiedPrompt` المُخزَّن في cache** — الـ `intentSignal` يُحقن في `buildRuntimeContext` وقت الـ generate step مباشرة (ليس في الـ cached prompt). هذا صحيح: الـ signal ديناميكي لكل رسالة بينما الـ prompt الأساسي ثابت ومُخزَّن. لا تحتاج لإبطال الـ cache عند تغيير intent.
9. **RESPONSE FORMATTING + GPT-4.1 nano** — قواعد التنسيق فعّالة بشكل خاص للنماذج الصغيرة. للنماذج الأكبر مثل GPT-4o، نفس القواعد تعمل مع هامش خطأ أصغر. لا أثر سلبي عند التبديل لنموذج أكبر مستقبلاً.
