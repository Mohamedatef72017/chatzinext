### 4. الأمان

- **Stripe webhook:** تحقق بـ `webhooks.constructEvent` — لا معالجة بدون توقيع.
- **Idempotency:** `PaymentEvent.stripeEventId` unique index يمنع معالجة نفس الحدث مرتين.
- **Checkout ownership:** التحقق من `sessionTenantId === tenantId` قبل إكمال أي checkout.
- **Wallet:** `$inc` atomic مع حماية صريحة من الرصيد السالب.
- **WalletTransaction:** `idempotencyKey` unique index يمنع التكرار.

### المهمة 3: تنفيذ استراتيجية Snapshot للخطة (ADR-001)

**الهدف:** ضمان أن تعديلات الخطط لا تؤثر على المشتركين الحاليين في منتصف دورة الفوترة.

**الملفات المعدّلة:** `src/lib/billing/plan-resolver.ts`, `src/lib/billing.ts`

**القرار المعماري (ADR-001):**
```
عند وجود TenantSubscription.planSnapshot.features[] وكان status ∈ {active, trialing, past_due}:
  → يستخدم الـ engine الـ snapshot المُجمَّد
  → تعديلات الخطة لا تؤثر فوراً على المشتركين

يُحدَّث الـ snapshot فقط عند:
  1. Checkout جديد (ترقية / تخفيض / اشتراك جديد) — billing.ts:handleCheckoutCompleted
  2. تجديد (invoice.payment_succeeded + billing_reason = subscription_cycle) — billing.ts:handleInvoicePaymentSucceeded
```

**التغييرات في `plan-resolver.ts`:**
- إزالة `populate("planId")` — لم يعد مطلوباً للمسار الرئيسي.
- إضافة `hasSnapshot && isActiveSubscription` check.
- إضافة `resolvedFrom: "snapshot" | "live_plan" | "default"` للـ debugging.
- Fallback للـ live plan عند غياب snapshot (مستأجرو الخطة المجانية / اشتراكات ملغاة).
- إضافة `source: "snapshot"` لكل ميزة مُحلَّلة من snapshot.

**التغييرات في `billing.ts:handleInvoicePaymentSucceeded`:**
- يقرأ الخطة الحالية عند كل تجديد.
- يُحدِّث `planSnapshot` في نفس write operation مع reset الـ usedMessages.
- التغييرات التي أجراها الأدمن على الخطة تدخل حيز التنفيذ في الدورة التالية.

**تأثير الأداء:** إيجابي — استبدال `populate("planId")` بـ `findOne` مباشر على subscription، ثم قراءة الـ snapshot من الـ document مباشرة دون join.

**تأثير الأمان:** محايد — الـ snapshot يمنع "price drift" غير المقصود بين دورات الفوترة.

**قابلية التوسع:** مرتفعة — الـ snapshot embedded في الـ document لا يتطلب query إضافية.

### المهمة 4: دورة حياة التجربة وفترة السماح (Cron Job)

**الهدف:** تطبيق آلي لانتهاء التجربة وفترة السماح بدون polling داخل الطلبات.

**الملفات الجديدة:** `src/app/api/cron/billing-lifecycle/route.ts`

**التصميم:**
- Endpoint HTTP آمن: `POST /api/cron/billing-lifecycle`
- مُؤمَّن بـ `CRON_SECRET` — يُرفض بدونه (401).
- يقبل `Authorization: Bearer <secret>` أو `x-cron-secret: <secret>`.
- Idempotent — آمن للتشغيل أكثر من مرة.
- قابل للربط بـ Vercel Cron, Upstash Scheduler, render.com cron, أو أي خدمة HTTP scheduler.

**Pass 1 — Trial Expiry:**
```
TenantSubscription.find({ status: "trialing", trialEndsAt: { $lte: now } })
→ لكل subscription:
    إذا كان stripeSubscriptionId موجوداً: يتحقق من Stripe → active إذا مدفوع، canceled إذا لا
    إذا لم يكن: canceled (لا طريقة دفع)
→ يُحدِّث status
→ يكتب SubscriptionHistory (trial_converted | trial_expired)
→ يكتب BillingAuditLog
```

**Pass 2 — Grace Period Expiry:**
```
TenantSubscription.find({ status: "past_due", graceEndsAt: { $lte: now } })
→ لكل subscription:
    يُلغي Stripe subscription إذا موجود (best-effort)
    يُحدِّث status → "canceled"
→ يكتب SubscriptionHistory (grace_expired)
→ يكتب BillingAuditLog
```

**GET endpoint:** يُعيد عدد الاشتراكات المنتظرة (للـ monitoring).

**متغيرات البيئة المطلوبة:**
```
CRON_SECRET=<secret_random_string>
```

**تكوين Vercel Cron (مثال في `vercel.json`):**
```json
{
  "crons": [
    { "path": "/api/cron/billing-lifecycle", "schedule": "0 */6 * * *" }
  ]
}
```

**تأثير الأمان:** عالٍ — لا يُعالَج أي طلب بدون `CRON_SECRET` صحيح.

**تأثير قابلية التوسع:** عالٍ — يعمل بـ single DB pass، لا loops داخل الطلبات الفردية، لا background threads.

## ملاحظات الترحيل

### شرط واحد قبل Deploy

```bash
# يجب تشغيله مرة واحدة على production database:
ts-node -r tsconfig-paths/register --compiler-options '{"module":"commonjs"}' scripts/seed-plan-features.ts
```

**لماذا؟** بعد حذف `PLAN_DEFAULTS_FALLBACK`، أي خطة بدون `features[]` ستُعيد `{}` من `plan-resolver`، مما يعني أن جميع الصلاحيات ستُرفض بـ `source: "default"`. الـ seed script آمن ومتعدد التشغيل.

### متغير البيئة الجديد


```json
}
```

1. **Dunning management:** `invoice.payment_failed` يُرسل إشعار email + retry logic قبل دخول grace period.
