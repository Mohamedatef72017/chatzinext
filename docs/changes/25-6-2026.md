## 11:30 AM — planSnapshot Strategy (ADR-001)

### Architecture Changes
- Defined ADR-001: subscriptions with `status ∈ {active, trialing, past_due}` resolve features from `TenantSubscription.planSnapshot.features[]` — not from the live `BillingPlan` document. The snapshot is updated only at checkout and at each renewal. Admin edits to a plan take effect on the next billing cycle, never mid-cycle.

### Backend Changes

**`src/lib/billing/plan-resolver.ts`**
- Removed `populate("planId")` — no longer needed for the primary resolution path.
- Added snapshot-first resolution: if `subscription.planSnapshot.features` exists and subscription is active/trialing/past_due, read features directly from the embedded snapshot without any extra DB query.
- Added fallback to live `BillingPlan.findById()` when no snapshot is present (free plan tenants, canceled subscriptions, new checkouts before first snapshot write).
- Added `resolvedFrom: "snapshot" | "live_plan" | "default"` field to `ResolvedPlan` — visible in debug/audit contexts.
- Added `source: "snapshot"` value to `ResolvedFeature.source` union.
- All three resolution paths (`snapshot`, `live_plan`, `default`) set local variables `planId`, `planName`, `planSlug`, `resolvedFrom` before the shared override pass.

**`src/lib/billing.ts`** (`handleInvoicePaymentSucceeded`)
- On each `invoice.payment_succeeded` with `billing_reason = subscription_cycle`, reads the current `BillingPlan` and writes a fresh snapshot into `TenantSubscription.planSnapshot`.
- Snapshot write is atomic with the `usedMessages` reset in the same `updateOne` call.
- **Impact**: Plan price/feature changes made by admins are guaranteed to activate on the next billing cycle. No retroactive changes to in-progress subscriptions.

### Database Changes
- No schema additions — `planSnapshot` field already exists on `TenantSubscription`. This change wires the read path for the first time.

### Performance Improvements
- Snapshot-first path replaces a `populate("planId")` join with a direct embedded-document read — eliminates one DB round-trip per `resolveTenantPlan()` call in the active-subscription case.

## 01:00 PM — Trial & Grace Period Lifecycle Cron

### Architecture Changes
- Moved trial-expiry and grace-period enforcement out of the hot request path and into a dedicated HTTP cron endpoint, safe for external scheduling with any HTTP-capable cron service (Vercel Cron, Upstash, render.com, etc.).

### Backend Changes

**`src/app/api/cron/billing-lifecycle/route.ts`** (new)
- `POST /api/cron/billing-lifecycle` — executes two idempotent passes:
  - **Pass 1 (Trial Expiry)**: finds all `{ status: "trialing", trialEndsAt: { $lte: now } }`. For each: verifies Stripe subscription status if `stripeSubscriptionId` present; transitions to `active` if Stripe confirms payment, otherwise `canceled`. Writes `SubscriptionHistory` (`trial_converted` or `trial_expired`) and `BillingAuditLog`.
  - **Pass 2 (Grace Expiry)**: finds all `{ status: "past_due", graceEndsAt: { $lte: now } }`. For each: cancels Stripe subscription (best-effort, continues on error); transitions to `canceled`; writes `SubscriptionHistory` (`grace_expired`) and `BillingAuditLog`.
- `GET /api/cron/billing-lifecycle` — returns `{ pendingTrials, pendingGrace }` counts for monitoring dashboards.
- Secured by `CRON_SECRET` env var — both `Authorization: Bearer <secret>` and `x-cron-secret: <secret>` headers accepted. Returns 401 without a valid secret.
- Fully idempotent — safe to run multiple times; already-processed subscriptions are skipped by the status+date query.
- Returns `{ ok, processedAt, trialExpired, graceExpired, errors }` for observability.
- **Impact**: Trial and grace enforcement runs on a predictable external schedule rather than relying on Stripe webhook delivery timing.

### New Environment Variables
```
CRON_SECRET=<random_string_min_32_chars>
```

### Example Vercel Cron Configuration (`vercel.json`)
```json
{
  "crons": [
    { "path": "/api/cron/billing-lifecycle", "schedule": "0 */6 * * *" }
  ]
}
```

## 02:30 PM — Prompt Cache Wiring

### Backend Changes

**`src/mastra/workflows/ai-reply.workflow.ts`**
- Added imports: `hashSettingsForPrompt`, `buildPromptCacheKey`, `getCachedSystemPrompt`, `setCachedSystemPrompt` from `src/lib/ai/prompt-cache.ts`.
- Updated `load-conversation` step — `unifiedPrompt` now:
  1. Computes a stable hash of all `AiSetting` fields that affect the system prompt.
  2. Builds a Redis cache key: `cache:prompt:{tenantId}:{botId}:{settingsHash}`.
  3. Returns cached prompt on hit.
  4. Builds prompt via `buildUnifiedSystemPrompt()` and writes to Redis cache on miss.
- Cache TTL: `AI_PROMPT_CACHE_TTL_SECONDS` (default 300 seconds).
- Cache key auto-invalidates when any of the following fields change: `systemPrompt`, `language`, `tone`, `tonePreset`, `role`, `responseLength`, `useEmojis`, `salesStyle`, `supportStyle`, `businessCategory`, `categoryPromptEn`, `customInstructionsEn`.
- **Impact**: `buildUnifiedSystemPrompt()` (string concatenation over multiple fields) runs once per `(bot, settings_version)` per 5 minutes instead of on every inbound message. For an active bot at 1,000 messages/hour this eliminates ~1,000 redundant string builds per hour.

## Risks / Notes

1. **`PLAN_DEFAULTS_FALLBACK` removal** — tenants whose `BillingPlan` document has no `features[]` array will receive an empty feature map (all entitlement checks denied). **Must** run `scripts/seed-plan-features.ts` on the production database before deploying.
2. **`CRON_SECRET` required** — if `CRON_SECRET` is not set, the cron endpoint returns 401 on every request. Set it before wiring the external scheduler.
3. **Prompt cache cold start** — first AI reply after deploy or after cache expiry rebuilds the prompt. This is expected behavior, not a failure.
4. **Stripe best-effort cancel on grace expiry** — if Stripe returns an error during grace expiry (e.g., subscription already canceled on Stripe side), the cron continues and marks the local record `canceled`. No retry mechanism for the Stripe call; Stripe state and local state should reconcile via the next webhook delivery.
5. **`resolvedFrom` is additive** — any TypeScript consumer of `ResolvedPlan` that uses an exhaustive type check on a subset of values will need updating; the new union member `"snapshot"` is now part of the type.
