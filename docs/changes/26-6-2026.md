## 09:00 AM — Response Formatting

### المشكلة

GPT-4.1 nano لا يُنسِّق الردود تلقائياً بدون توجيه صريح — يدمج أفكاراً متعددة في كتلة نص واحدة، ولا يستخدم قوائم bullet عند عرض الخيارات أو الخدمات.

| الإمكانية | GPT-4o | GPT-4.1 nano |
|---|---|---|
| الالتزام بالتنسيق بدون توجيه صريح | ✅ | ❌ |
| استخدام قوائم بدون أمر | ✅ | ⚠️ غير موثوق |
| فصل الأفكار في فقرات | ✅ | ❌ |

### Architecture Changes

إضافة قسم تنسيق كامل (RESPONSE FORMATTING) في `GLOBAL_CRM_SYSTEM_PROMPT` — يُطبَّق على جميع التهيئات وجميع النماذج.

### Backend Changes

**`src/lib/ai/build-system-prompt.ts`**

أُضيف قسم **RESPONSE FORMATTING** مباشرةً بعد قسم RESPONSE QUALITY في `GLOBAL_CRM_SYSTEM_PROMPT`.

القواعد الخمس المُضافة:

```
SEPARATE YOUR IDEAS:
كل فكرة مستقلة في فقرتها الخاصة مع سطر فارغ بين الفقرات.
ممنوع دمج أفكار متعددة في كتلة نص واحدة.

USE LISTS WHEN APPROPRIATE:
- 3 عناصر أو أكثر (خدمات، أسعار، خطوات، خيارات) → قائمة bullet، عنصر لكل سطر
- تسلسل خطوات → قائمة مُرقَّمة (1. → 2. → 3.)
- ممنوع ضغط القائمة في جملة واحدة بفواصل

KEEP PARAGRAPHS SHORT:
2–3 جمل لكل فقرة كحدٍّ أقصى.

NEVER SEND A WALL OF TEXT:
إذا تجاوز الرد 4 أسطر بدون سطر فارغ → كثيف جداً، اقسمه.

DO NOT USE MARKDOWN DOCUMENT SYNTAX:
ممنوع ## أو ### headers أو horizontal rules.
هذه واجهة محادثة وليست مستنداً.
```

- **التأثير**: ردود منظمة ومقروءة على جميع النماذج. الفائدة الأكبر على النماذج الصغيرة التي تحتاج توجيهاً صريحاً.

### Risks / Notes

- **زيادة حجم `GLOBAL_CRM_SYSTEM_PROMPT`** — الـ prompt cache يُخفف الأثر. يُبنى الـ prompt مرة واحدة لكل `(bot, settings_version)` كل 5 دقائق.

## 10:30 AM — Intent Signal Injection

### المشكلة

النموذج يستنتج حالة العميل (جاهز للشراء / حساس عاطفياً) من نص قصير — وهو استنتاج ضعيف في النماذج الصغيرة. الرسالة "عايز أحجز موعد" كانت تحصل أحياناً على رد يُعيد شرح الخدمات بدل بدء خطوات الحجز مباشرةً.

### Architecture Changes

حقن إشارة سلوكية صريحة (`intentSignal`) في `buildRuntimeContext` بعد اكتشاف النية من `detectBusinessIntent`. يصل هذا الـ signal للنموذج عبر `contextSummary` في system prompt.

```
[knowledgeStep]
      ↓
detectBusinessIntent(message) → "appointment" | "complaint" | ...
      ↓
[generateReplyStep]
      ↓
buildRuntimeContext(inputData)
      ↓
contextSummary = "businessIntent=appointment; intentSignal=purchase_ready → ..."
      ↓
buildUnifiedSystemPrompt({ contextSummary })
      ↓
Runtime context section في system prompt
```

### Backend Changes

**`src/mastra/workflows/ai-reply.workflow.ts`**

تحديث import:
```typescript
import {
  detectBusinessIntent,
  isDirectKnowledgeIntent,
  isPurchaseReadyIntent,   // جديد
  isHighSensitivityIntent, // جديد
  type BusinessIntent      // جديد
} from "@/lib/ai/business-intent";
```

إضافة حقن الـ signal في `buildRuntimeContext` مباشرةً بعد `businessIntent`:
```typescript
if (inputData.businessIntent) {
  parts.push(`businessIntent=${inputData.businessIntent}`);

  const intent = inputData.businessIntent as BusinessIntent;

  if (isPurchaseReadyIntent(intent)) {
    parts.push(
      "intentSignal=purchase_ready → customer is close to booking or buying; " +
      "skip re-explaining services; move directly to the next concrete action; " +
      "close warmly and concisely"
    );
  } else if (isHighSensitivityIntent(intent)) {
    parts.push(
      "intentSignal=high_sensitivity → customer needs careful emotional handling; " +
      "lead with empathy and acknowledgment before anything else; " +
      "do NOT upsell or cross-sell; prioritize resolution over all else"
    );
  }
}
```

**التأثير على ردود النموذج:**

| الرسالة | intent | intentSignal | السلوك المتوقع |
|---|---|---|---|
| "بكام الخدمة؟" | `prices` | `purchase_ready` | يُجيب بالسعر + خطوة الحجز مباشرة |
| "عايز أحجز موعد" | `appointment` | `purchase_ready` | يبدأ خطوات الحجز فوراً |
| "عندي مشكلة" | `support` | `high_sensitivity` | يبدأ بالإقرار العاطفي، لا upsell |
| "عندي شكوى" | `complaint` | `high_sensitivity` | تعاطف أولاً، حل ثانياً، لا بيع |
| "إيه الخدمات؟" | `services` | *(لا signal)* | سلوك عادي |

- **التأثير**: النموذج يتصرف بشكل صحيح من أول رسالة بدون الحاجة لاستنتاج ضعيف من سياق قصير.

### Risks / Notes

- **intentSignal ديناميكي** — يُحقن في `buildRuntimeContext` وقت generate step مباشرة وليس في الـ cached prompt. هذا صحيح: النية تتغير لكل رسالة، بينما الـ prompt الأساسي ثابت. لا تحتاج لإبطال الـ cache عند تغيير intent.

## 03:30 PM — Bug Fix 2: Stripe Key Validation

### المشكلة

عند محاولة الترقية يظهر خطأ: `"Invalid API Key provided: sk_test_***mhKW"`.

`makeStripe()` تمرر المفتاح مباشرة لـ Stripe بدون أي تحقق من صيغته — إذا كان المفتاح منتهي الصلاحية أو من بيئة خاطئة، تأتي رسالة الخطأ من Stripe مباشرةً بدون أي سياق تشخيصي مفيد.

### Backend Changes

**`src/lib/billing/providers/stripe.provider.ts`**

إضافة فحص صيغة المفتاح في `makeStripe()`:

```typescript
// قبل
function makeStripe(): Stripe {
  const key = process.env.STRIPE_SECRET_KEY;
  if (!key) throw new Error("STRIPE_SECRET_KEY is required.");
  return new Stripe(key);
}

// بعد
function makeStripe(): Stripe {
  const key = process.env.STRIPE_SECRET_KEY;
  if (!key) throw new Error("STRIPE_SECRET_KEY is required.");
  if (!key.startsWith("sk_test_") && !key.startsWith("sk_live_")) {
    throw new Error(
      "STRIPE_SECRET_KEY is invalid: key must start with sk_test_ or sk_live_. " +
      "Verify that the correct Stripe secret key is configured in environment variables. " +
      `Current key prefix: ${key.slice(0, 8)}...`
    );
  }
  return new Stripe(key);
}
```

- **التأثير**: رسالة خطأ واضحة عند تكوين مفتاح Stripe بصيغة خاطئة بدل انتظار رد Stripe الغامض.

### خطوات تشخيص البيئة

إذا استمر الخطأ بعد هذا الإصلاح، المشكلة في المفتاح المُكوَّن نفسه:

1. تحقق من `STRIPE_SECRET_KEY` في environment variables — يجب أن يبدأ بـ `sk_test_` (للتطوير) أو `sk_live_` (للإنتاج)
2. تحقق من صلاحية المفتاح في [Stripe Dashboard → API Keys](https://dashboard.stripe.com/apikeys)
3. تحقق من أن المفتاح يتبع الـ Stripe account الصحيح وليس account منتهي الصلاحية

## 04:30 PM — Bug Fix 4: Prompt Cache Utilization

### المشكلة

كل رسالة واردة تستدعي `buildUnifiedSystemPrompt()` مرتين:

```
load-conversation step:
  → buildUnifiedSystemPrompt() → يُخزَّن في Redis cache
  → يُخزَّن في inputData.unifiedPrompt ✅

generate-reply step:
  → buildUnifiedSystemPrompt() من الصفر ← الكود القديم يتجاهل inputData.unifiedPrompt ❌
```

الـ cache الذي يُبنى في `loadConversationStep` لم يكن يُستخدَم أبداً في `generateReplyStep`.

### Architecture Changes

فصل بناء الـ prompt إلى جزأين:
- **الجزء الثابت** (static): businessName، botName، role، tone، language، customInstructions — يُبنى مرة واحدة في `loadConversationStep` ويُخزَّن في Redis cache ثم في `inputData.unifiedPrompt`
- **الجزء الديناميكي** (dynamic): knowledgePrompt، runtimeContext، needsLeadInfo — يُلحَق في `generateReplyStep` لكل رسالة

### Backend Changes

**`src/mastra/workflows/ai-reply.workflow.ts`** — دالة `generateReplyStep`

```typescript
// قبل: بناء كامل في كل رسالة (يتجاهل الـ cache)
const instructions = [
  buildUnifiedSystemPrompt({
    businessName: ...,
    botName: ...,
    role: ...,
    tone: ...,
    // ... كل الحقول
    knowledgeInstructions: inputData.knowledgePrompt, // ديناميكي
    contextSummary: runtimeContext,                    // ديناميكي
    needsLeadInfo: inputData.needsLeadInfo,            // ديناميكي
  }),
].filter(Boolean).join("\n\n");

// بعد: استخدام الـ cache + إلحاق الأجزاء الديناميكية فقط
const basePrompt = inputData.unifiedPrompt || buildUnifiedSystemPrompt({
  businessName: ...,
  botName: ...,
  role: ...,
  // ... الحقول الثابتة فقط (بدون knowledge/context/needsLeadInfo)
});
const instructions = [
  basePrompt,
  inputData.needsLeadInfo
    ? "CRM FIELD COLLECTION ACTIVE: ..."
    : "",
  inputData.knowledgePrompt
    ? `Available business knowledge for this conversation:\n${inputData.knowledgePrompt}`
    : "",
  runtimeContext
    ? `Runtime context:\n${runtimeContext}`
    : "",
].filter(Boolean).join("\n\n");
```

### Performance Impact

| السيناريو | قبل | بعد |
|---|---|---|
| Cache hit (الحالة الشائعة) | 2 × `buildUnifiedSystemPrompt()` | 1 × (في load step فقط) |
| Cache miss (أول رسالة أو تغيير settings) | 2 × | 1 × build + كتابة cache |
| بوت نشط بـ 1,000 رسالة/ساعة | ~2,000 string builds | ~1,000 string builds |

- **التأثير**: يُلغي استدعاء `buildUnifiedSystemPrompt()` الزائد في كل رسالة. الـ cache الذي كان يُبنى ويُهمَل الآن يُستخدَم فعلاً.

### Risks / Notes

- عند **cache miss** (أول رسالة بعد 5 دقائق من الهدوء)، يستخدم الـ fallback ويُعيد البناء — سلوك صحيح.
- الأجزاء الديناميكية (knowledgePrompt، runtimeContext، needsLeadInfo) لا تُخزَّن في cache — هذا مقصود؛ تتغير مع كل رسالة.

## مشاكل تتطلب تدخلاً في البيئة (بدون تعديل كود)

### قاعدة المعرفة لا تُستخدَم (Bug 1)

**التحقيق:** Pipeline الـ AI يبحث في KB بشكل صحيح وتكتيكياً سليم. السبب الأرجح للأعراض المُبلَّغة:

**سيناريو A — Tenant جديد بلا documents مُدرَّبة:**
إذا لم تُرفع مصادر KB بعد، أو المصادر لا تزال بـ status `"pending"` (تنتظر training)، تُعيد `searchKnowledge` نتائج فارغة. النموذج يتلقى:
```
"No specific business knowledge was found for this customer message."
```
ويُجيب بشكل عام.

**سيناريو B — fast-intent-responder يُعالج رسائل الهوية:**
الرد "أنا ChatZi، المساعد الافتراضي..." هو رد **صحيح ومقصود** من `fast-intent-responder` على سؤال هوية ("من أنت؟"). ليس تجاوزاً للـ KB — KB لا تُستخدَم أصلاً لأسئلة الهوية، وهذا السلوك الصحيح.

**خطوات التشخيص:**
```bash
# تحقق من وجود documents مُدرَّبة
db.knowledgedocuments.find({ tenantId: "...", status: "approved" }).count()

# تحقق من وجود chunks
db.knowledgechunks.countDocuments({ tenantId: "...", botId: "..." })

# إذا كانت documents بـ status "pending" → شغّل إعادة التدريب
POST /api/knowledge/retrain
```

**تتبع logs الإنتاج:**
```
ai.knowledge_retrieval → ragResults > 0 يعني KB تُستخدَم
ai.knowledge_retrieval → ragResults = 0 يعني لا chunks للبوت هذا
```
