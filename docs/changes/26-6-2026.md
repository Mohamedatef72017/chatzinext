# ChatZi Engineering Change Log — 26 June 2026

## Summary

تحسينان متكاملان مُصمَّمان خصيصاً لتعزيز أداء **GPT-4.5 nano** (النموذج الخفيف المستخدم في الإنتاج):

1. **Response Formatting** — قسم تنسيق كامل في `GLOBAL_CRM_SYSTEM_PROMPT` يُلزم النموذج بردود مقروءة ومنظمة — فقرات منفصلة، قوائم bullet ومُرقَّمة، لا كتل نص متراصة.
2. **Intent Signal Injection** — حقن إشارة سلوكية صريحة في `buildRuntimeContext` ترشد النموذج للسلوك الصحيح (شراء / حساسية عاطفية) بدلاً من الاستنتاج الضعيف من السياق.

| الإمكانية | GPT-4o | GPT-4.5 nano |

## التغيير الأول — Response Formatting

### الملف: `src/lib/ai/build-system-prompt.ts`

أُضيف قسم **RESPONSE FORMATTING** جديد في `GLOBAL_CRM_SYSTEM_PROMPT` مباشرةً بعد قسم RESPONSE QUALITY.

**القواعد الخمس المُضافة:**

**SEPARATE YOUR IDEAS:**
كل فكرة مستقلة في فقرتها الخاصة مع سطر فارغ بين الفقرات. ممنوع دمج أفكار متعددة في كتلة نص واحدة.

**USE LISTS WHEN APPROPRIATE:**
- 3 عناصر أو أكثر (خدمات، أسعار، خطوات، خيارات) → قائمة bullet point، عنصر واحد لكل سطر
- تسلسل خطوات يتبعها العميل → قائمة مُرقَّمة (1. → 2. → 3.)
- ممنوع ضغط القائمة في جملة واحدة بفواصل

**KEEP PARAGRAPHS SHORT:**
2–3 جمل لكل فقرة كحدٍّ أقصى. إذا احتاج الموضوع تفصيلاً، يُقسَّم إلى فقرات قصيرة متعددة.

**NEVER SEND A WALL OF TEXT:**
إذا تجاوز الرد 4 أسطر بدون سطر فارغ → كثيف جداً، اقسمه.

**DO NOT USE MARKDOWN DOCUMENT SYNTAX:**
ممنوع ## أو ### headers أو horizontal rules. هذه واجهة محادثة وليست مستنداً.

**المكان في الملف:** بعد سطر `Do not write the same closing line in every message. Vary your language naturally.` وقبل قسم KNOWLEDGE COMPLIANCE.

## التغيير الثاني — Intent Signal Injection

### الملف: `src/mastra/workflows/ai-reply.workflow.ts`

#### تحديث الـ Import

```typescript
// قبل
import { detectBusinessIntent, isDirectKnowledgeIntent } from "@/lib/ai/business-intent";

// بعد
import {
  detectBusinessIntent,
  isDirectKnowledgeIntent,
  isPurchaseReadyIntent,
  isHighSensitivityIntent,
  type BusinessIntent
} from "@/lib/ai/business-intent";
```

#### تحديث `buildRuntimeContext`

إضافة حقن الـ signal مباشرةً بعد `businessIntent`:

```typescript
if (inputData.businessIntent) {
  parts.push(`businessIntent=${inputData.businessIntent}`);

  const intent = inputData.businessIntent as BusinessIntent;

  if (isPurchaseReadyIntent(intent)) {
    parts.push(
      "intentSignal=purchase_ready → customer is close to booking or buying; " +
      "skip re-explaining services; move directly to the next concrete action; " +
      "close warmly and concisely"
    );
  } else if (isHighSensitivityIntent(intent)) {
    parts.push(
      "intentSignal=high_sensitivity → customer needs careful emotional handling; " +
      "lead with empathy and acknowledgment before anything else; " +
      "do NOT upsell or cross-sell; prioritize resolution over all else"
    );
  }
}
```

#### كيف يصل الـ signal للنموذج

```
[knowledgeStep]
      ↓
detectBusinessIntent(message) → "appointment" | "complaint" | ...
      ↓
[generateReplyStep]
      ↓
buildRuntimeContext(inputData)
      ↓
contextSummary = "businessIntent=appointment; intentSignal=purchase_ready → ..."
      ↓
buildUnifiedSystemPrompt({ contextSummary })
      ↓
Runtime context section in system prompt — يراه GPT-4.1 nano
```

#### التأثير المباشر على ردود GPT-4.5 nano

| الرسالة | intent | intentSignal | السلوك المتوقع |
|---|---|---|---|
| "بكام الخدمة؟" | `prices` | `purchase_ready` | يُجيب بالسعر + خطوة الحجز مباشرة |
| "عايز أحجز موعد" | `appointment` | `purchase_ready` | يبدأ خطوات الحجز فوراً |
| "عندي مشكلة" | `support` | `high_sensitivity` | يبدأ بالإقرار العاطفي، لا upsell |
| "عندي شكوى" | `complaint` | `high_sensitivity` | تعاطف أولاً، حل ثانياً، لا بيع |
| "إيه الخدمات؟" | `services` | *(لا signal)* | سلوك عادي |

## Verification Steps

### Response Formatting
1. اطلب من البوت سرد الخدمات → يجب أن يكون الرد قائمة bullet، وليس جملة واحدة بفواصل.
2. اطلب خطوات الحجز → يجب أن تكون الإجابة قائمة مُرقَّمة 1. 2. 3.
3. رسالة سؤال عام به نقطتان منفصلتان → تحقق من وجود سطر فارغ بينهما.
4. `grep -n "RESPONSE FORMATTING" src/lib/ai/build-system-prompt.ts` → يجب أن يُعيد نتيجة.

### Intent Signal Injection
1. أرسل رسالة "عايز أحجز موعد" → تحقق من أن context الـ agent يحتوي `intentSignal=purchase_ready`.
2. أرسل رسالة "عندي شكوى" → تحقق من أن context يحتوي `intentSignal=high_sensitivity`.
3. أرسل رسالة "إيه الخدمات؟" → تحقق من عدم وجود `intentSignal` في context.
4. `isPurchaseReadyIntent("appointment")` → `true`. `isPurchaseReadyIntent("services")` → `false`.
5. `isHighSensitivityIntent("complaint")` → `true`. `isHighSensitivityIntent("prices")` → `false`.

## Risks / Notes

1. **intentSignal لا يُعاد حقنه في `unifiedPrompt` المُخزَّن في cache** — الـ `intentSignal` يُحقن في `buildRuntimeContext` وقت الـ generate step مباشرة وليس في الـ cached prompt. هذا صحيح: الـ signal ديناميكي لكل رسالة بينما الـ prompt الأساسي ثابت ومُخزَّن. لا تحتاج لإبطال الـ cache عند تغيير intent.
2. **RESPONSE FORMATTING + GPT-4.5 nano** — قواعد التنسيق فعّالة بشكل خاص للنماذج الصغيرة. للنماذج الأكبر مثل GPT-4o، نفس القواعد تعمل مع هامش خطأ أصغر. لا أثر سلبي عند التبديل لنموذج أكبر مستقبلاً.
3. **زيادة حجم `GLOBAL_CRM_SYSTEM_PROMPT`** — إضافة RESPONSE FORMATTING رفعت الـ prompt لـ ~200 سطر. Prompt cache يُخفف الأثر: الـ prompt يُبنى مرة واحدة كل 5 دقائق لكل `(bot, settings_version)`.

### Bug 1 — أسماء العلامات التجارية تُعرَّب/تُنقحر (Bug 2 في القائمة الأصلية)

**الأعراض:** البوت يكتب "شاتزي" بدل "ChatZi" و"مواتف" بدل "Moatef" عند الرد بالعربية.

BRAND NAME RULE:
Never translate, transliterate, or convert brand names, business names, bot names,
or product names into a different script or language. Write them exactly as configured
— character for character. If the bot name is "ChatZi", write "ChatZi" — never "شاتزي".
If the business name is "Moatef", write "Moatef" — never "مواتف". This rule is absolute
and applies in every language including Arabic, English, or any other.

**الملف المُعدَّل:** `src/lib/ai/build-system-prompt.ts`

### Bug 2 — `aiPausedReason` / `handoffReason` غامض (Bug 3 في القائمة الأصلية)

**الأعراض:** عند إيقاف البوت بسبب طلب إنساني، يُخزَّن في DB `handoffReason: "ticket_required_fields_complete"` أو `"crm_ticket_ai_policy_engine"` — أسباب غامضة تُصعِّب التشخيص ولا تعكس السبب الحقيقي.

**السبب الجذري:** `persistResultStep` في `ai-reply.workflow.ts` عند تنفيذ handoff يستخدم:
```typescript
aiPausedReason: inputData.reason || "ticket_created",
handoffReason: inputData.reason || "ticket_created",
```
وقيمة `inputData.reason` تأتي من محرك ticket flow (أسباب تقنية داخلية) لا من السبب الفعلي للإيقاف.

**ملاحظة:** منطق `shouldHandoff` صحيح — الإيقاف يحدث فقط عند `ticket.shouldCreate === true && ticket.category === "human_request"` (أي طلب صريح للإنسان). الخطأ في التسمية فقط.

**الإصلاح:** استخدام قيمة ثابتة وواضحة:
```typescript
aiPausedReason: "human_handoff_requested",
handoffReason: "human_handoff_requested",
```

**الملف المُعدَّل:** `src/mastra/workflows/ai-reply.workflow.ts` — دالة `persistResultStep`، السطرين 887 و889

### Bug 3 — فشل الفوترة بـ "Invalid API Key" (Bug 4 في القائمة الأصلية)

**الأعراض:** عند محاولة الترقية، يظهر خطأ "Invalid API Key provided: sk_test_***mhKW".

**السبب الجذري المحتمل (كود):** `makeStripe()` في `stripe.provider.ts` لا يتحقق من صيغة المفتاح قبل تمريره لـ Stripe. إذا كان المفتاح المُكوَّن في env vars بصيغة خاطئة (منتهي الصلاحية، مقطوع، أو key من بيئة مختلفة)، يصل الخطأ من Stripe مباشرةً بدون رسالة تشخيصية مفيدة.

**السبب الجذري المحتمل (بيئة):** `STRIPE_SECRET_KEY` في environment variables خاطئ أو منتهي الصلاحية. يجب التحقق من Stripe Dashboard.

**الإصلاح (كود):** إضافة validation للصيغة في `makeStripe()`:
```typescript
function makeStripe(): Stripe {
  const key = process.env.STRIPE_SECRET_KEY;
  if (!key) throw new Error("STRIPE_SECRET_KEY is required.");
  if (!key.startsWith("sk_test_") && !key.startsWith("sk_live_")) {
    throw new Error(
      "STRIPE_SECRET_KEY is invalid: key must start with sk_test_ or sk_live_. " +
      "Verify that the correct Stripe secret key is configured in environment variables. " +
      `Current key prefix: ${key.slice(0, 8)}...`
    );
  }
  return new Stripe(key);
}
```

**الملف المُعدَّل:** `src/lib/billing/providers/stripe.provider.ts`

**خطوات التشخيص للبيئة:**
1. تحقق من `STRIPE_SECRET_KEY` في environment variables — يجب أن يبدأ بـ `sk_test_` (للتطوير) أو `sk_live_` (للإنتاج)
2. تحقق من صلاحية المفتاح في [Stripe Dashboard → API Keys](https://dashboard.stripe.com/apikeys)
3. تحقق من أن المفتاح يتبع Stripe account الصحيح (ليس account مختلف أو منتهي الصلاحية)

### Bug 4 — `generateReplyStep` يُعيد بناء الـ prompt كاملاً متجاهلاً الـ cache (Bug 5 في القائمة الأصلية)

**الأعراض:** أداء أبطأ من المتوقع. كل رسالة تستدعي `buildUnifiedSystemPrompt` مرتين: مرة في `loadConversationStep` (مع cache) ومرة في `generateReplyStep` (بدون cache).

**السبب الجذري:** `loadConversationStep` يبني `unifiedPrompt` ويُخزِّنه في `inputData.unifiedPrompt`. لكن `generateReplyStep` يتجاهل هذه القيمة ويستدعي `buildUnifiedSystemPrompt` مجدداً بشكل كامل.

**الإصلاح:** استخدام `inputData.unifiedPrompt` كـ base في `generateReplyStep` وإلحاق الأجزاء الديناميكية فقط (knowledge + context + needsLeadInfo):

```typescript
// قبل: بناء كامل في كل مرة
const instructions = [buildUnifiedSystemPrompt({...all fields...})].filter(Boolean).join("\n\n");

// بعد: استخدام cache + dynamic suffix
const basePrompt = inputData.unifiedPrompt || buildUnifiedSystemPrompt({...static fields only...});
const instructions = [
  basePrompt,
  needsLeadInfo ? "CRM FIELD COLLECTION ACTIVE: ..." : "",
  knowledgePrompt ? `Available business knowledge:\n${knowledgePrompt}` : "",
  runtimeContext ? `Runtime context:\n${runtimeContext}` : "",
].filter(Boolean).join("\n\n");
```

**الفائدة:** عند تطابق settings التهيئة (الحالة الشائعة)، يُحذَف استدعاء `buildUnifiedSystemPrompt` بالكامل في `generateReplyStep`. الـ cache في `loadConversationStep` يُستخدَم فعلاً الآن.

**الملف المُعدَّل:** `src/mastra/workflows/ai-reply.workflow.ts` — دالة `generateReplyStep`، السطور 650-675

### Bug 5 — قاعدة المعرفة لا تُستخدَم / رد عام دائماً (Bug 1 في القائمة الأصلية)

**التحقيق:** بعد مراجعة كامل pipeline:
1. `knowledgeStep` يبحث في KB بشكل صحيح (مُصفَّى بـ tenantId + botId)
2. `buildKnowledgePrompt` يُمرِّر النتائج أو رسالة "no knowledge" للـ LLM
3. `generateReplyStep` يُمرِّر `knowledgeInstructions` للـ system prompt بشكل صحيح

**السبب الأرجح للأعراض المُبلَّغة:**
- **Tenant جديد بلا documents**: إذا لم يُرفَع أي مصدر للـ KB أو المصادر لا تزال بـ status `"pending"` (تحتاج training)، فـ `searchKnowledge` يُعيد نتائج فارغة. الـ LLM يتلقى رسالة "No specific business knowledge was found" ويُجيب بشكل عام.
- **fast-intent-responder يُعالج رسائل الهوية**: رسالة "أنا شاتزي، المساعد الافتراضي..." هي رد **صحيح** من `fast-intent-responder` على سؤال هوية ("من أنت؟"). هذا سلوك مقصود — لكن أسماء العلامات التجارية كانت تُعرَّب (مُصلَح في Bug 1 أعلاه).

**خطوات التشخيص:**
1. تحقق من وجود documents في KB: `KnowledgeDocument.find({ tenantId, status: "approved" })`
2. تحقق من chunks: `KnowledgeChunk.countDocuments({ tenantId, botId })`
3. إذا كانت المصادر بـ status `"pending"` → trigger retraining: `POST /api/knowledge/retrain`
4. تتبع logs `ai.knowledge_retrieval` للتأكد من `ragResults` و`confidence`

**لا يوجد تعديل كود مطلوب** — المشكلة بيانات لا كود.
