ChatZi Enterprise Billing Refactor — Implementation Prompt

You are a Principal SaaS Architect, Senior Backend Engineer, Billing Systems Engineer, and Security Engineer.

Your task is to implement a major billing architecture refactor for ChatZi based on the billing audit findings.

The goal is to transform the current billing system from a hardcoded AI-message-based billing system into a generic, configuration-driven, feature-based SaaS billing platform.

You MUST work from the actual source code only.

Do NOT rewrite the entire project.

Do NOT remove existing working Stripe functionality.

Do NOT break existing tenants.

Do NOT introduce temporary hacks.

Do NOT make unrelated changes.

Every change must preserve:

* Performance & Scalability
* Security
* Availability
* Maintainability
* Clean Code
* Backward Compatibility
* Tenant Isolation

⸻

Core Problem To Fix

Currently, billing depends heavily on:

* aiMessageLimit
* monthlyMessageLimit
* usedMessages
* extraMessageCredits
* PLAN_DEFAULTS
* hardcoded entitlement keys
* Stripe-specific logic inside billing service
* non-generic quota logic

This makes it hard to add future billable features like:

* AI tokens
* Knowledge storage
* File storage
* Channel limits
* Telegram channel count
* Messenger page count
* WhatsApp number count
* Team seats
* API calls
* Workflow executions
* Voice minutes
* OCR pages
* Attachment storage
* Model-specific AI pricing

The target architecture must be generic and feature-driven.

⸻

Target Architecture

Implement a clean billing domain model based on:

Feature Definition

A reusable feature that can be attached to any plan.

Examples:

* ai_messages
* ai_tokens
* knowledge_storage_mb
* attachment_storage_mb
* team_members
* agents
* bots
* channels_total
* telegram_channels
* messenger_channels
* whatsapp_channels
* facebook_channels
* instagram_channels
* api_requests
* workflow_runs
* custom_domain
* white_label
* advanced_ai
* qdrant_search
* audit_logs

Each feature must support:

* key
* name
* description
* type: boolean | quota | count | storage | metered
* unit
* resetPeriod: monthly | yearly | never
* isActive
* metadata

⸻

Plan Feature

Each plan must contain its own features and limits.

A plan must become the single source of truth.

The system must not depend on hardcoded PLAN_DEFAULTS.

Example:

features: [
  {
    key: "ai_messages",
    type: "quota",
    limit: 10000,
    resetPeriod: "monthly",
    overageAllowed: false
  },
  {
    key: "telegram_channels",
    type: "count",
    limit: 2
  },
  {
    key: "knowledge_storage_mb",
    type: "storage",
    limit: 10240
  },
  {
    key: "advanced_ai",
    type: "boolean",
    enabled: true
  }
]

⸻

Subscription

Subscription must reference:

* tenantId
* planId
* planVersionId or planSnapshot
* status
* currentPeriodStart
* currentPeriodEnd
* cancelAtPeriodEnd
* trialEndsAt
* graceEndsAt
* provider
* providerCustomerId
* providerSubscriptionId
* usage summary

The subscription should not be the only place where limits live.

Limits come from the plan or plan snapshot.

⸻

Usage Engine

Create a generic usage engine.

Replace usage logic that only supports AI messages.

Required generic APIs:

getFeatureLimit(tenantId, featureKey)
canUseFeature(tenantId, featureKey, amount)
reserveUsage(tenantId, featureKey, amount, options)
commitUsage(tenantId, featureKey, amount, options)
releaseUsage(tenantId, featureKey, amount, options)
getUsageSnapshot(tenantId)
resetUsageForPeriod(tenantId, period)

Usage must support:

* monthly reset
* yearly reset
* never reset
* Redis atomic counters
* MongoDB persistence
* idempotency keys
* sync from Redis to MongoDB
* failure handling
* tenant isolation
